PM MasterMind — Outreach Timing Architecture

BIA Feature G · 2026-06-23 · 7 advisors · 3 peer reviewers · 1 chairman

The Challenge

BIA shipped Feature G — "Outreach Timing Windows" — as a free wedge for 5 future list-building features. 72 personas × ~8 trigger cards each (576 total). Each card: Day 0 anchor, optimal day range, why-this-window, too-early / too-late, and a "Recommended move."

Founder Barry caught after ship that the "Recommended moves" hallucinate product-specific framings (e.g., "OCC examination readiness brief authored by a former OCC examiner on your advisory board" / "FIS Horizon certified integration case study") that mislead any seller whose product doesn't match. 576 cards live in production.

Parallel persistent bug: Cards refer to personas by character names ("Riley", "Sarah") instead of role ("the Banking CIO"). Flagged repeatedly across features.

Three options on the table:

  • A — Strip product specifics. Shape-only moves. ~$1.40 regen. Stays free/static/instant.
  • B — Dynamic paid per-user. $0.02/view, 8–15s, becomes paid.
  • C — Hybrid. Static scaffold + live one-line move using productContext. $0.01/view cached.

Council Verdict Chairman

Where the MasterMind Agrees

Despite the surface 4–3 vote (A vs C), 7 of 7 advisors converged on the same diagnosis: the team shipped 576 cards with no telemetry, no validated activation event, and Barry as the only test surface. The architecture debate is downstream of the discovery debate.

  • You have no chain of evidence. Tharin: "Option A doesn't lack telemetry because it's static — it lacks telemetry because nobody wired it." Lenny: "Do you have a retention curve for ANY free surface?"
  • Activation event is unvalidated. Is the aha moment viewing the card or running the play? No one has answered this.
  • The character-name leak is a process gap. Perri: "Recurring across features = no systemic learning loop." A template lint / persona schema gate fixes it; A/B/C does not.
  • Fix Riley/role first, regardless of architecture path.
  • This pattern propagates to 5 future features. The discovery and instrumentation contract set here will set the precedent for C, E, F, B.

Where the MasterMind Clashes

The 4–3 split maps to a deeper disagreement about which loop matters most:

  • A camp (Verna, Perri, Torres, Tharin): The wedge function depends on shareability and trust. A canonical artifact requires a universal form. Per-user personalization breaks the artifact, kills shareability, and pre-$15M ARR makes any meter on a wedge = PLS-too-early.
  • C camp (Lenny, Bush, Cagan): The wedge function depends on behavior change. Shape-only is PLG 1.0 regression — forces the AE to translate. C makes productContext load-bearing, which is also the upgrade hook.

Until BIA measures both shareability and behavior change, this argument can't be resolved.

Blind Spots Peer Review Caught

  1. Trust blast radius. 576 cards with fabricated product claims are live to paying users — customer-trust incident, not just architecture. No advisor proposed in-app changelog or user heads-up.
  2. The Riley/role bug is a generation-pipeline integrity signal. Same pipeline produced all 576 cards. Has anyone validated the WINDOWS themselves — Day 0 anchors, day ranges, rationale? The moat is unaudited.
  3. Verna's "Growth Engine" thesis is unevidenced. "Shareable B2B asset" was pattern-matched onto a feature with zero observed sharing behavior.

Priority Fixes

1
Ship Option A today as the tourniquet — but only after fixing the character-name leak and adding a single in-app instrumentation event. Regenerate all 72 timing files with a prompt that forbids (a) character first names, (b) named product examples, (c) seller-side capability claims. Anchor every Recommended move to a buyer-side artifact. ~$1.40, ~5 min.
2
Add a template-review gate before merging any persona-touching generator. Define the persona-rendering contract and enforce it as a generation-time guardrail. Stops the next 5 features from shipping the same class of bug.
3
Run a 48-hour qualitative test before committing to Option C. Strip 10 cards manually, show 5 AEs stripped + 5 current, ask "what would you do with this?" In parallel, flag-test draft C to 20 users. Decide on C in 30 days based on data.

The One Thing to Do First

Rewrite the generate-timing-windows.js prompt with three explicit constraints: (a) role only, never character names, (b) no product specifics or vendor capability claims, (c) anchor every Recommended move to a buyer-side artifact (MRA timeline, asset tier peer reference, regulatory milestone). Regenerate all 72 files. Deploy. Stops both bugs in one pass. Then wire the telemetry hook.

Advisor Responses

Lenny Rachitsky — Retention/Loop votes C

Stop. Before picking A/B/C, answer this: do you have a retention curve for any existing free surface? Podcasts, Buyer Intel Docs — what's the W2/W4 return rate? If you don't know, you're flying blind on this entire decision.

The activation event isn't "viewing a card." It's "running a play and reporting back." A timing window the user never acts on is a vanity activation — same failure mode as a dashboard nobody opens twice. The question is whether Feature G creates a loop: see card → send the move → trigger fires or doesn't → come back to log/learn/iterate. None of A/B/C addresses the loop.

Retention risk ranking: A is highest. Shape-only moves are the exact generic advice experienced AEs already ignore. They'll view once, shrug, never return. That's your leaky bucket. B is lowest retention risk but worst activation friction. C is the only one that preserves instant + specific + free-feeling.

Ship C. Instrument it for the loop, not the view. Cache by user-product-persona-trigger, fix the Riley/Sarah role bug in the same regen, add a "did you run this?" check-back the following week. Talk to 5 AEs who viewed a card 7 days ago. If <40% acted, it's a PMF problem on the wedge, not a regen problem.

Elena Verna — Racecar / Growth Engine votes A

Option A. Strip the product specifics. Ship shape-only moves.

Racecar read: the 576 cards are a Growth Engine candidate — content-driven acquisition loop. Shareable artifact + structured proprietary data + screenshot-native format = compounding loop substrate.

Option B kills the engine to chase ARPU. Per-user dynamic = no canonical artifact to share. Converts a compounding loop into a Turbo Boost. It's Fake PLG — paywalls the acquisition surface.

Option C inherits B's flaw: the Recommended move is the screenshot-worthy line. Personalize it and shareability collapses.

Option A keeps the engine intact. Shape-only moves — "Send a regulator-aligned readiness brief positioned around your differentiated control" — are honest, universal, and screenshottable. Product-specific layer belongs in paid surfaces (Positioning Playbook, Cold Email Sequence). Fix the "Riley" naming bug in the same pass. Engine integrity is non-negotiable.

Wes Bush — PLG / Time-to-Value votes C

The Aha moment is #3 — acting on the Recommended move. Everything upstream is scaffolding.

Option A is Fake PLG. Stops one step before the activation event and asks the user to translate. "Lead with peer reference matching asset tier" is PLG 1.0 — the user does the synthesis. You moved the hallucination problem onto the user.

Option B violates the Halving Principle on a surface that's currently instant. 8-15s on a free wedge feature trains users that the product is slow before they've experienced value.

Option C. Genuine PLG 2.0. Static scaffolding preserves instant TTV. The Recommended move regenerates once per user-product-persona-trigger and caches forever. AI does the translation work, at $0.01 amortized.

Two conditions: (1) Fix the Riley/role leak first. (2) Pre-warm the cache for the user's top 3 personas during onboarding completion. Don't make activation wait on a cold cache.

Melissa Perri — Build Trap votes A

Yes, BIA is in the Build Trap on Feature G. 576 cards shipped, hallucinations caught by the founder after production — that's not discovery, that's QA-by-Barry. Measured output (cards generated), not outcome (does the seller act?).

The recurring character-name bug is the louder signal. No systemic learning loop, no shared definition of "done," no pre-ship checklist that encodes prior founder feedback. Textbook Product Death Cycle.

Option C is the seductive build-trap answer. More infrastructure feels like progress, but you're scaling a feature whose outcome you haven't validated.

Pick A. $1.40 regen, ships today, stops the bleeding, lets you run the actual experiment: do sellers report that timing windows changed their outreach behavior? Instrument one question in-app after card view. If <20% say yes, no version of B or C matters.

Teresa Torres — Discovery / Opportunity Solution Tree votes A (as tourniquet)

You're choosing between three solutions without an opportunity map.

What's the outcome? Is Outreach Timing supposed to drive return visits, screenshot loops, or trust in the data layer? Each points to a different option. You haven't named the outcome, so you can't rank the options.

The opportunity space is unmapped. "I don't know when to call" wants Day 0 + window. "I don't know what to say without burning the relationship" wants the Recommended move. "I don't want to look stupid screenshotting this" wants shape-only. Three different unmet needs, three different winners.

Cheapest experiment: Strip 10 cards (1 hour, zero cost). Show 5 AEs the stripped version and 5 the current version. Ask: "What would you do with this?" Know in 48 hours.

Pick A now as the bug-fix tourniquet — hallucinations on a trust surface are bleeding. Then run the interview before committing engineering to B or C.

Marty Cagan — Four Big Risks votes C with guardrails

Pick C. But not for the reasons Barry's framing suggests.

The Four Big Risks expose the real problem: nobody validated Value before committing 576 cards. Feature Factory. Barry's own post-ship "this feels wrong" is the only data point. One PM, one tester, zero discovery.

Value: A is honest but generic. C wins on value if productContext is rich enough. Untested.
Usability: A makes the AE do the translation. C reduces friction at the moment of action.
Feasibility: C's cache invalidation is real but bounded.
Viability: Keep the wedge free. Grounded Recommended move is the upgrade hook — requires onboarding (productContext), which deepens activation.

Verdict: Ship C behind a flag to 20 users. Measure: did they act on the move? Kill the character-name leak first. Missionary, not mercenary.

Leah Tharin — PLG/PLS architecture / Instrumentation votes A

Pick A — with non-negotiable instrumentation built before the next card ships.

You're treating "free vs paid" as the decision when the actual decision is "do you have a chain of evidence from card-open to simulator-entry to credit-burn." Right now you have none. Option A doesn't lack telemetry because it's static — it lacks telemetry because nobody wired it.

Five events to wire: card opened, time-on-card, copy-clicked, "Open in Simulator" clicked, simulator session within 30min of card view, persona-match between card and session.

Option C is seductive because per-view generation logs itself by construction — but that's lazy instrumentation dressed up as product strategy. The moment you meter it, you've built a soft paywall at pre-$15M ARR. Textbook PLS-too-early.

Define the activation event as simulator session opened from a card within 30 minutes, same persona. Re-evaluate in 30 days. If cards drive simulator entry, you've got your wedge. If not, no personalization saves it.
